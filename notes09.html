Why Commit Messages Matter

  • Employers will review commit messages in your github when you apply.
  • Commit messages will allow other developers on a project to look specifically through your changes when a bug arises.
  • Good CMs will allow you to pick up a project where you left off after time away.
  • A diff will tell you what changed, but only the commit message can properly tell you why.

How to Ensure good commit messages

    Before beginning a project a team should agree to a convention for CMs that defines at least the
  • Style. Markup syntax, wrap margins, grammar, capitalization, punctuation. Spell these things out, remove the guesswork, and make it all as simple as possible.
  • Content. What should the messages contain, and what should they not contain?
  • Metadata. How should issue tracking IDs, pull request numbers, etc. be referenced?

The Seven Rules of a Great Git Commit Message

  1. Separate subject from body with a blank line
  2. Limit the subject line to 50 characters
  3. Capitalize the subject line
  4. Do not end the subject line with a period
  5. Use the imperative mood in the subject line
  6. Wrap the body at 72 characters
  7. Use the body to explain what and why vs. how
1. Subject and Body

Some commits only require a subject, such as "fix typo in opening paragraph of "About" page.

Having a logline separated from the body makes it easy to print out just the subject using commands in a CLI, such as "git shortlog" and "git log --oneline".

CMs longer than just a subject are difficult to type using the -m condition in the command line, and should be typed out in a text editor.

2. 50 characters

Not a hard rule, but forces author to keep it concise and readable

Git warns if over 50, caps at 72

3. Capitalize

Just the first letter of the subject line

It keeps it looking nice and emphasizes the convention is being used

4. No period in subject line

Unnecessary, especially with limited characters

5. Imperative mood

As if giving a command

"Fix typo in opening paragraph" vs "Fixed typo in opening paragraph"

This convention cuts down on characters, forces more direct thinking and communication, and follows the convention used by git autocommit

Think "If applied, this commit will YOUR COMMIT SUBJECT LINE HERE"

ie. "If applied this commit will fix the typo in the opening paragraph"

6. Wrap at 72 chars

Git doesn't auto wrap at the edge - 80 characters.

Wrapping manually at 72cs allows git to fit with an indent at 80

Text editors can assist in wrapping and be set to auto wrap CM bodies.

7. What and Why vs How

Code is generally self explanatory as to how.

If code change (what) isn't self explanatory given the why, use a source comment to justify the how.
